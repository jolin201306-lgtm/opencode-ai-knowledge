# AI 知识库 · 编码规范 · 参考版 v1.0

> Week 1 · 第 2 节 · B 路产出示例
> 这是跑完 grill-me 8 轮拷问后的终态 · 你自己的版本可能重点不同

## 要做什么

- **Python 格式**：black · `line-length=100`（默认 88 对中文注释太挤）
- **TypeScript**：`strict: true` + `noUncheckedIndexedAccess: true`
- **文档**：所有公开函数必须有文档
  - Python 用 Google 风格 docstring
  - TS 用 TSDoc · `@param` / `@returns` 必填
  - private 函数不强制 · 但命名必须自解释
- **行宽统一**：Python 100 · TS 100
- **导入顺序**：Python isort · 分 stdlib / 3rd-party / local 三组
- **命名规约**：Python `snake_case` · TS `camelCase` · 类 `PascalCase` · 文件 `kebab-case`
- **AI 生成标识**：AI 生成的代码必须有 `# Generated by <agent-name> on <date>` 注释
- **AI 修改 commit**：AI 修改过的文件 commit message 用 `[AI]` 前缀

## 不做什么

- 不用魔法字符串（必须提到模块顶部 `UPPER_SNAKE` 常量）
  - 例外：单次使用的用户消息、测试数据
- 不允许 TODO / FIXME / XXX / HACK 直接进 main（只允许 NOTE）

## 边界 & 验收

- **单测覆盖率分层**：
  - 业务逻辑层（`services/`、`handlers/`）≥ 90%
  - 工具层（`utils/`、`helpers/`）≥ 80%
  - 边界适配层（`adapters/`、`clients/`）≥ 50%（mock 难写）
  - 总平均 ≈ 80%
- **lint 检查工具链**：
  - Python：black + ruff + mypy
  - TypeScript：eslint + tsc strict

## 怎么验证

- `make lint` 一键跑全部格式检查（black / ruff / eslint）
- CI 每个 PR 跑 lint + 单测覆盖率检查
- `make test-coverage` 输出分层覆盖率报告
- pre-commit hook 拦截不符合规范的 commit（Week 2 N5 实现）
- `[AI]` commit 必须经过人工 review 才能 merge

## grill-me 追问的 8 轮演化

这份 spec 不是一次写成的。grill-me 依次拷问了：

1. black 行宽是 88 还是 100？ → 100
2. TypeScript strict mode 具体哪几个？ → strict + noUncheckedIndexedAccess
3. docstring 风格？ → Google + TSDoc
4. 魔法字符串边界？ → 例外给单次用户消息
5. TODO 一起禁吗？ → 禁 TODO/FIXME/XXX/HACK · 留 NOTE
6. 覆盖率哪 80%？ → 分层（业务/工具/适配）
7. 行宽 / 导入顺序 / 命名规约没提 → 补齐
8. AI 专属约束（生成标识 + commit 前缀） → 加

你的项目可能重点不同（比如只有 Python 没有 TS），产出条款数量会不一样。这份只是参考。
